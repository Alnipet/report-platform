# Описание архитектуры: Report Platform

Этот документ содержит ключевые решения при проектировании платформы отчетов согласно требованиям тестового задания.

## 1. Архитектура: Компоненты, Потоки Данных, Границы

Платформа разделена на следующие компоненты:
1. **Frontend (React / SPA)**: Отвечает за взаимодействие с пользователем. Запрашивает доступные отчеты, инициирует генерацию и пуллит статусы (в будущем может быть заменен на WebSockets).
2. **API Backend (NestJS)**: Принимает REST запросы. Играет роль `Gateway` — создает запись в БД (статус PENDING) и кладет `JobId` в `Redis Queue`.
3. **Queue / Message Broker (BullMQ + Redis)**: Инфраструктурный слой для надежной асинхронной обработки задач с поддержкой ретраев.
4. **Worker (NestJS Module)**: В данном прототипе запущен в том же процессе, что и API, для упрощения (в рамках `docker-compose`). Отвечает за: получение задачи из очереди -> нахождение нужного отчета в реестре -> генерацию -> сохранение файла в Storage -> обновление статуса в БД.
5. **Database (PostgreSQL + Prisma)**: Хранит историю (таблица `ReportRun`).
6. **Storage**: Временно локальная папка (подмонтирована как Docker Volume). В Production это будет интерфейс, за которым скрыт S3 (MinIO, AWS S3).

**Поток данных**:
`[User] -> (POST /request) -> [API] -> (DB Insert) -> (Redis LPUSH) -> [User получает RunID]`
`(Redis BRPOP) -> [WorkerProcessor] -> [BaseReportGenerator.generate()] -> [Storage Save] -> (DB Update COMPLETED) -> [User скачивает по URL]`

## 2. DX: Как добавить новый отчёт за 1 рабочий день

Разработчику команды, чтобы добавить новый отчет (например, "Статистика логов" в PDF), нужно сделать всего **два шага**:

1. Создать класс, унаследованный от `BaseReportGenerator` (см. `backend/src/reports/generators/base-report.generator.ts`).
2. Зарегистрировать его в `reports.module.ts`.

**Пример:**
```typescript
@Injectable()
export class LogsSummaryReport extends BaseReportGenerator {
  id = 'logs-summary';
  name = 'Статистика логов (PDF)';
  format = 'pdf';

  async generate(context: ReportContext): Promise<Buffer> {
    // 1. Пойти в Clickhouse / Kibana / Внутреннее API (занять 2-3 часа на написание SQL/Fetch)
    const data = await this.myDb.query('...');

    // 2. Собрать PDF с помощью pdfmake/pdfkit (занять 2 часа)
    const pdfBuffer = await MyPdfFormatter.format(data);

    // 3. Вернуть Buffer
    return pdfBuffer;
  }
}
```

## 3. Принятые решения и альтернативы

### Решение 1: Асинхронность через Redis + BullMQ
**Почему**: Генерация XLSX/PDF блокирует Event Loop Node.js. Запускать это в основном HTTP-потоке запрещено (это положит API для других клиентов). BullMQ — стандарт де-факто, надежный, имеет ретраи и планировщик.
**Альтернативы**: RabbitMQ (излишен для прототипа, сложнее в настройке), Kafka (Overkill для платформы такого размера), просто массив `setTimeout` (потеря данных при падении сервера).

### Решение 2: NestJS для бэкенда
**Почему**: Паттерн Registry (Реестр отчетов) идеально реализуется через Dependency Injection. NestJS имеет строгую структуру. В команде легко навигироваться по модулям.
**Альтернативы**: Express.js (пришлось бы писать свой IOC-контейнер и архитектуру слоев).

### Решение 3: Monolith vs Microservices (в рамках прототипа)
**Почему**: В `docker-compose` запущен один сервис `api` (модульный монолит). BullMQ-воркер и HTTP-сервер крутятся в одном NestJS приложении. Это сделано намеренно для экономии оперативной памяти на демо-стенде и ускорения развертывания.
**Альтернативы**: Разнести API Gateway и Worker-Processing-Node в два разных микросервиса. В Production-окружении (Kubernetes) мы бы так и сделали, чтобы независимо скейлить Worker'ы в зависимости от длины очереди (HPA).

## 4. Что не сделали и что добавить для продакшена

Поскольку продукт позиционируется как MVP для проверки подхода, следующие "продакшен" вещи заменены стабами:

1. **Storage (Хранилище)**:
   *Сейчас*: Локальная директория `/tmp/reports`.
   *Для прода*: Внедрить абстракцию `IStorageService` и адаптер `S3StorageAdapter` (aws-sdk).
2. **Аутентификация и авторизация (RLS)**:
   *Сейчас*: Отчеты может генерировать кто угодно по HTTP.
   *Для прода*: Добавление JWT Guards в NestJS. В таблице `ReportRun` должно храниться `userId`. Сделать Row-Level Security, чтобы пользователь мог скачать только свой отчет.
3. **WebSockets (Server-sent events)**:
   *Сейчас*: React-клиент делает Polling раз в 3 секунды.
   *Для прода*: Socket.IO / SSE для пуш-уведомлений клиенту, что отчет готов (сохраняет ресурсы сети).
4. **Stream Generation**:
   *Сейчас*: PDF и XLSX полностью генерируются в `Buffer` в памяти сервера (в ОЗУ).
   *Для прода*: Если отчет на 250MB, Buffer убьет V8. Необходимо изменить возвращаемый тип метода `generate()` с `Promise<Buffer>` на `Promise<ReadableStream>` и писать напрямую в S3 multipart-загрузкой.
